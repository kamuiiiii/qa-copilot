# /add-target — 新建 target

用户输入：$ARGUMENTS（可选，可以是 target 名字，例如 `admin`）

target 文件的字段结构权威定义见 `targets/example.md`。本命令只做"问 + 填模板 + 落盘"。

## 执行 SOP（严格按顺序）

### 1. 拿到并校验 target 名

- 如果 `$ARGUMENTS` 给了，直接用；否则问：
  > 要新建的 target 叫什么名字？（小写英文/数字/`-`，会作为文件名 `targets/<name>.md`）
- **校验**：只允许 `[a-z0-9-]`，且不以 `-` 开头/结尾。不合法就让用户重起一个。

### 2. 检查是否已存在

如果 `targets/<name>.md` 已经存在，**不要直接覆盖**。问用户：

> `targets/<name>.md` 已存在，要 (1) 覆盖 / (2) 改名 / (3) 取消？

按用户选择处理；选 (3) 直接结束这次命令，不写文件。

### 3. 交互式问 2 个问题（一次一个）

按顺序问，每问一个等用户回答。`URL` 必填，`环境` 可以回 `skip` 跳过——跳过的字段保留 example 里的占位提示，让用户后续手动补。

| # | 问题                                                          | 是否必填 |
| - | ------------------------------------------------------------- | -------- |
| 1 | 这个系统的 URL 是什么？（多套环境就写最常测的，比如 staging） | 必填     |
| 2 | 是 qa / staging / pre-prod / prod 哪种？                      | 可 skip  |

> 登录方式不用问——本项目所有被测系统统一走 OAuth，由 QA 在 Chrome 里手动登录（见 `CLAUDE.md` 的"浏览器接入"）。

### 4. 写文件

- 读 `targets/example.md` 拿到模板结构。
- 把 2 个回答按顺序塞进 **基本信息** 的对应 bullet。
- skip 的字段保留 example 里的占位文字，让用户后续手动补。
- 写到 `targets/<name>.md`。

### 5. 收尾汇报

告诉用户：

- 文件路径 `targets/<name>.md`
- 哪些字段被 skip 还是占位状态（如果有）
- 一句使用提示：以后跑测可以说"针对 `targets/<name>` 跑 ..."

最后问一句：

> 要现在打开这个文件再细调一下吗？

用户回 yes 就用 `Read` 展示内容，让他们说哪里要改。
